\title{\textbf{Longest Increasing Subsequence Problem in Small Space, revisited\\}}

\date{\today}
\author{Mikołaj Owczarz}

\begin{document}

\maketitle

\begin{abstract}
Given an input array of comparable elements, we are to find either a longest increasing subsequence, or its length using parametrised space.
It is well known that this problem can be solved in $\mathcal{O}(n \log n)$ time without constrained memory. For parametrised memory -- $s$, where
$\sqrt{n} \leq s \leq n$ we present an improved analysis of space parametrised $Patience\ Sorting$ algorithm presented in \cite{kiyomi2017space}. We show that the length computation runs in $\mathcal{O}(\operatorname{max}(n \log n, \frac{n^2}{s}))$ time.
We also show an improved Divide \& Conquer algorithm for obtaining the sequence running in $\mathcal{O}(\operatorname{max}(n \log n \log \log n, \frac{n^2}{s}))$ time while using $\mathcal{O}(s \log n)$ bits.
\end{abstract}

\section{Introduction}

The longest increasing subsequence (LIS for short) problem is a classic algorithmic problem. 
The input is a sequence of $n$ numbers $A = (a_1, a_2, \dots, a_n)$, and we are to either find the length of LIS -- 
the largest integer $k$ such that there exist indices $i_1 < i_2 < \dots < i_k$ such that $a_{i_1} < a_{i_2} < \dots < a_{i_k}$, or to return this subsequence.

\subsection{Previous results}
Efficient algorithms for finding the LIS are known, for example the Patience Sorting algorithm devised by Mallows \cite{mallows} that runs in $\mathcal{O}(n \log n)$ for a sequence of length $n$ while using $\mathcal{O}(n)$ words in working space.
Given a sequence of $n$ elements, \Psort\ assigns each element to a $pile$, where each pile contains an information about the longest increasing subsequence of the input that ends in that element. 
While the \Psort\ algorithm achieves the lower bound in terms of time complexity for finding a LIS, the question of how little memory one needs to efficiently solve the problem is still unanswered. 
In \cite{kiyomi2017space} we can see a variation of \Psort\ that for given $s$ can find the length of a LIS in $\mathcal{O}(\frac{n^2}{s} \log n)$ and compute the actual subsequence in $\mathcal{O}(\frac{n^2}{s} \log^2 n)$, but only for $\sqrt{n} \leq s$.

\subsection{Our contribution}
In this paper we improve the bounds achieved by \cite{kiyomi2017space}. For finding the length of a LIS we conduct a tighter time analysis showing that the proposed in \cite{kiyomi2017space} algorithm actually runs in $\mathcal{O}(\operatorname{max}(n \log n, \frac{n^2}{s}))$ time,
while maintaining the $\mathcal{O}(s)$ bound for the number of words in the working memory. 
Next we observe that an alternate, simpler Divide \& Conquer algorithm can be derived, and is more efficient than one provided in \cite{kiyomi2017space}.
Combined with the tighter analysis for LIS length computation it runs in $\mathcal{O}(\operatorname{max}(n \log n \log \log n, \frac{n^2}{s}))$.

\subsection{Related work}
The LIS problem, besides the classic setting, has many heavily-researched variations.
For example we could ask for the longest increasing subsequence only one some part of the input by making range queries (\cite{range-lis}).
We can also ask about maintaining the LIS in dynamic setting (\cite{dynamic-lis}, \cite{dynamic-lis-lb}). Another interesting question is 
approximation of LIS in any of this setting (\cite{approx-lis}, \cite{approx-lcs}).

\section{Preliminaries}

We define two variants of the Longest Increasing Subsequence problem.

\problembox
{Longest increasing Subsequence (\Lis)}
{A sequence of $n$ numbers $A = (a_1, a_2, \dots, a_n)$.}
{The longest subsequence $(a_{i_1}, a_{i_2}, \dots, a_{i_k})$ of $A$ such that $i_1 < i_2 < \dots < i_k$ and $a_{i_1} < a_{i_2} < \dots < a_{i_k}$.}

\problembox
{Lenght of \Lis \ (\LisLen)}
{A sequence of $n$ numbers $A = (a_1, a_2, \dots, a_n)$.}
{$k$ -- the largest integer such that there exist an increasing subsequence in $A$ of length $k$}


We use the RAM model, where the input sequence is read-only, and the working memory is limited to $\mathcal{O}(s)$ words.

\subsection{Patience Sorting}

Our algorithm is based on the classic \Psort. Let us define basic notions used throughout the algorithm. The idea of the algorithm is maintaining $Piles$\ -- stacks of integers such that for all $x \in A$, $x$ belongs to the $i$-th pile if and only if the $longest\ increasing\ subsequence$ of $A$ ending in $x$ has length $i$.
Since \Psort\, both classic, and space-parametrised is based on sequential passes through the input sequence we define
pile operations with respect to the time when they are being used.

\begin{definition}
For integers $i, t$ we refer to the state $i$-th pile after reading first $t$ elements of the input sequence as $P_{i}^{t}$, and $P_i$ will represent the $i$-th pile after reading the whole input sequence. \\
Let $P$ be some state of the pile (for example $P_3^4$ or $P_2$).
\begin{itemize}
\item $|P|$ is the number of elements that belongs to $P$
\item $\operatorname{top}(P)$ refers to the element on the top of the pile (the one that was added most recently to it), or null if $P$ is empty.
\item For integer $j$, $P[j]$ is the $j$-th element of $P$, where $\operatorname{top}(P) = P[|P|]$, and $P[1]$ is the element on the bottom of $P$. 
\end{itemize}
\end{definition}

It follows from the pile definition that if we are solving the \LisLen\ problem, then our goal is to compute the largest integer $k$ such that $P_k$ is non-empty.
Classic \Psort\ algorithm achieves that by constructing all piles during a single passing through the input sequence. When reading the $t$-th element of $A$ -- $a_t$, we find
the smallest $i$ such that $\operatorname{top}(P_i^{t-1}) \geq a_t$ and we push $a_t$ to the top of the $i$-th pile, or we create a new, rightmost pile with a single element $a_t$.
Notice that by definition each element belongs to exactly one pile, thus for every $t$, if $a_t$ is in $i$-th pile then $P_j^{t-1} = P_j^{t}$ if $j \neq i$ and $P_i^t$ is $P_i^{t-1}$ with $a_t$ pushed on top.
Because the difference in states of piles changes in a tractable way, we omit the upper indexes in the implementation of \Psort\ algorithm below.

\begin{algorithm}
    \caption{\Psort}
    \begin{algorithmic}[1]
        \State initialize $P_{0}$ with the single element $-\infty$
        \State set $k := 0$ \Comment{\textcolor{gray}{$k$ will be the index of the rightmost non-empty pile}}
        \For{$a_t \in A$}
            \If{$a_t > \text{top} (P_{k})$}
                \State create an empty pile -- $P_{k+1}$, and push $a_t$ to $P_{k+1}$
                \State set $k := k + 1$
            \Else
                \State set $h$ to be the smallest index with $x \leq \operatorname{top}(P_h)$, and push $x$ to $P_h$
            \EndIf
        \EndFor
        \State \Return $k$
  \end{algorithmic}
\end{algorithm}


\begin{observation}\label{obs:pile}(\cite{burnstein})
This procedure correctly computes piles, in other words: \\
Element $a_t \in P_i$ if and only if a $longest\ increasing\ subsequence$ of $A$ ending in $a_t$ is of length $i$
\end{observation}

\subsection{Space parametrised Patience Sorting}

Now we will define the space parametrised variant of \Psort\ that originates from \cite{kiyomi2017space}.

Assume that our working memory is limited to $\mathcal{O}(s)$ words, and $\sqrt{n} \leq s \leq n$.
Similarly to the classic \Psort\ we will process the input array sequentially while maintaining piles. However, since our memory is limited and we cannot afford
to keep all piles in the memory, we will pass through the input multiple times. Each time we will memoise only one, chosen pile and some top elements of other piles. The idea is clearly presented in \cite{kiyomi2017space} (Lemma 4.2, Lemma 4.3).

More precisely, now we will define a procedure \Ppass\ that given the $i$-th pile $P_i$ performs two passes through the input array.
During the first pass it computes an index $j$ such that $i + s \leq j \leq i + 2s \land |P_j| \leq s$ or $j$ is \LisLen. 
During the second pass, now with given $P_i$ and $j$ it will compute whole $P_j$, or only the $\operatorname{top}(P_j)$ if $j$ is \LisLen.

\begin{observation}\label{obs:filtering} (\cite{kiyomi2017space}, Observation 4.1)
Suppose we are reading the $t$-th element of $A$ -- $a_t$, and $a_t \in P_j$. Let $i$ be some fixed integer. \\
If $j \neq i$, then 
\begin{itemize}
\item $a_t < \operatorname{top}(P_i^{t-1})$ if $j < i$
\item $a_t > \operatorname{top}(P_i^{t-1})$ if $j > i$
\end{itemize} 
\end{observation}

Recall that in \Psort\ procedure, for an element $a_t$, to determine to which pile it belongs it suffices to know only values of the top elements of piles in moment $t-1$. 
Moreover, by Observation~\ref{obs:filtering} if we want to check if this index -- $k$ belongs to some interval, in our case we want to check if $i \leq k \leq i + 2s$ it suffices to know only top elements 
of the piles whose indexes are also in this interval. Below we define the $filtering$ method for the \Ppass\ procedure. \\

Suppose we are processing $a_t$. We can encounter one of these three cases: 
\begin{enumerate}
\item If $a_t = \operatorname{top}(P_i^t)$, then $k = i$.
\item Otherwise, if $a_t < \operatorname{top}(P_i^{t-1})$, then by Observation~\ref{obs:filtering}, $k < i$ and we ignore $a_t$.
\item Otherwise $a_t > \operatorname{top}(P_i^{t-1})$, and again by Observation~\ref{obs:filtering}, $k > i$. \\
Let $r := \operatorname{top}(P_{i + 2s}^{t-1})$, then:
    \begin{enumerate}
    \item if $r$ exists and $a_t > r$, then $k > i + 2s$ and we ignore $a_t$.
    \item else if $r$ is undefined or $a_t \leq r$, we have $i < k \leq i + 2s$, and $k$ is the smallest index such that $a_t \leq \operatorname{top}(P_k^{t-1})$.
    \end{enumerate}
\end{enumerate}
Observe that if the first case is false, then $\operatorname{top}(P_i^{t-1}) = \operatorname{top}(P_i^{t})$ thus exactly one of these cases holds. \\

Notice, that if we have whole $P_i$ in the memory we can process first two cases in time $\mathcal{O}(1)$ by maintaining a pointer to the most recently read element on $P_i$ -- $h = \operatorname{top}(P_i^{t-1})$.

In the first case we compare $a_t$ to the element in $P_i$ that is after $h$. We can determine if we are in the second or third case by comparing $a_t$ and $h$.

It is easy to see, that if we maintain $\operatorname{top}(P_k^{t-1})$ for all $i < k \leq i + 2s$ then we can distinguish cases 3(a) and 3(b) in $\mathcal{O}(1)$ and find the value of $k$ in 3(b) in $\mathcal{O}(\log s)$ time.
We will call the search in 3(b) an $expensive\ filter$. Below we present the \Ppass\ implementation (\cite{kiyomi2017space} Algorithm 3).

\begin{algorithm}[H]
    \caption{First pass in \Ppass, assume $P_i$ is given, and $|P_i| < s$}
    \begin{algorithmic}[1]
    \State set $h := 0$ \Comment{\textcolor{gray}{$h$ points to the most recently read element in $P_{i}$}}
    \State set $r := i$ \Comment{\textcolor{gray}{Is the index of the current rightmost pile}}
    \State set $p_{i+1}, p_{i+2}, \ldots, p_{i+2s}$ to $\infty$ \Comment{\textcolor{gray}{$p_k$ is the current $\operatorname{top}(P_{k})$}}
    \State set $c_{i+1}, c_{i+2}, \ldots, c_{i+2s}$ to $0$ \Comment{\textcolor{gray}{$c_k$ is the current size of $P_k$}}
    \For{$ x \in A$} 

        \If{$x$ is $P_i[h+1]$} \Comment{\textcolor{gray}{$x$ is the next element in $P_i$}}
            \State set $h := h + 1$ 
            \State continue
        \ElsIf {$x < P_i[h]$ or $x > p_{i+2s}$} \Comment{\textcolor{gray}{$x$ belongs to the $P_k$ where $k < i \vee k > i+2s$}}
            \State continue
        \EndIf

        \If{$x > p_r$}
            \State set $r := r + 1$ and $k := r$
        \Else
            \State set $k$ to be the smallest such that $x < p_k$
        \EndIf
            
        \State set $p_k := x$ and $c_k := c_k + 1$
    \EndFor
    
    \State set $j$ to be the biggest, such that $0 < c_j \leq s$
    
    \Return $j$
    \end{algorithmic}
\end{algorithm}

After this pass, if value of $r < i + 2s$ we can conclude that $r$ is \LisLen\ since no element belongs to a pile with larger index.
Then we can set $j := r$ and return $j$ as \LisLen\ and $p_j = \operatorname{top}(P_j)$.  
It is easy to see how to modify the procedure to compute the whole $P_j$ if $j$ is given, we can maintain additional stack and push elements they are in the $j$-th pile.


\begin{observation} \label{obs:ppass-advance}
Since $s > \sqrt{n}$ there is at most $\frac{n}{s} \leq s$ piles with more than $s$ elements which implies that $j > i + s$ 
\end{observation}

\begin{observation} \label{obs:ppass-time}
For single \Ppass\, the time complexity can be written as $\mathcal{O}(n + W \log s )$, where $W$ is the number of calls to the expensive filter during the procedure.
\end{observation}

\section{New results}

The complexity analysis for the space parametrised \Psort\ in \cite{kiyomi2017space} for finding \LisLen\ yielded
$\mathcal{O}(\frac{n^2}{s} \log n)$ time. By tighter estimation of the number of calls to the expensive filter procedure defined before we are able to prove, that in fact this algorithm runs in $\mathcal{O}(\operatorname{max}(n \log n, \frac{n^2}{s}))$.
It was also presented in \cite{kiyomi2017space} how to compute the \Lis\ by using Divide \& Conquer algorithm with time complexity of $\mathcal{O}(\frac{n^2}{s} \log^2 n)$. We provide a simpler and more effective Divide and Conquer strategy that combined with improved bound for computing the \LisLen\ runs in $\mathcal{O}({\operatorname{max}(n \log n \log \log n, \frac{n^2}{s})})$ time.


\subsection{Length computation}
We provide a tighter time complexity analysis of the \Psort\ algorithm presented in \cite{kiyomi2017space}. 
We show that the algorithm runs in $\mathcal{O}(\operatorname{max}(n \log n, \frac{n^2}{s}))$ time, improving the previous $\mathcal{O}(\frac{n^2}{s} \log n)$ bound.

\begin{definition}
    Let $j_i$ be the index of the pile kept in memory after $i$-th \Ppass\ with $j_0=0$.
\end{definition}

\begin{definition}
Let $W_i$ be the number of $expensive\ filter$ operations during the $i$-th \Ppass.
\end{definition}

\begin{definition}
Let $K$ be the total number of \Ppass -es.
\end{definition}

By using Observation~\ref{obs:ppass-time} we can see that time complexity of space parametrised \Psort\ for $\sqrt{n} \leq s \leq n$ can be then written as:
\[
T(n,s) \in \mathcal{O}(nK + \log s \sum_{i=1}^K W_i).
\]

\begin{claim} \label{claim:linear-filters}
The total number of expensive filter operations is linear, i.e.,
\[
\sum_{i=1}^K W_i \in \mathcal{O}(n).
\]
\end{claim}

\begin{proof}
For each pass $i \in \{1, \dots, K\}$, let us define the active interval $I_i = (j_{i-1}, j_{i-1}+2s]$.

Fix a pass $i$. During the first phase of this pass, the $expensive\ filter$ is invoked only for elements whose pile index belongs to $I_i$. Once the new boundary $j_i$ is determined, the second phase invokes the $expensive\ filter$ only if the pile index is in $(j_{i-1}, j_i]$. 
By \Ppass\ construction, we know that $j_i \leq j_{i-1} + 2s$, which means $(j_{i-1}, j_i] \subseteq I_i$. Thus, we can see that:
\[
W_i \le \sum_{k \in I_i} |P_k| + \sum_{k=j_{i-1}+1}^{j_i} |P_k| \leq 2\sum_{k \in I_i} |P_k|.
\]
This gives us: 
\[
\sum_{i=1}^K W_i \le 2\sum_{i=1}^K \sum_{k \in I_i} |P_k|.
\]

By \Ppass\ construction (Observation~\ref{obs:ppass-advance}) we have $j_i \ge j_{i-1}+s$. This implies that the left endpoints of the intervals $I_i$ advance by at least $s$ in each pass. Since each interval $I_i$ has a width of exactly $2s$, any pile index $k$ can belong to at most two intervals $I$. Therefore, summing the pile sizes over all intervals yields at most twice the total number of elements:
\[
\sum_{i=1}^K \sum_{k \in I_i} |P_k| \le 2\sum_{k=1}^n |P_k| = 2n.
\]
Thus, we get $\sum_{i=1}^K W_i \le 4n \in \mathcal{O}(n)$.
\end{proof}

Because after each \Ppass\ we advance at least by $s$ piles and the largest possible pile index is $n$, $K \in \mathcal{O}(\frac{n}{s})$, which gives us the total running time:
\[
T(n,s) \leq cn \frac{n}{s} + c'n\log s
\]
For some constants $c, c'$. It is easy to see that $T(n, s) \in \mathcal{O}(\operatorname{max}(n\log n, \frac{n^2}{s}))$ as desired.

\subsection{LIS computation}
To \Lis, we present an alternative Divide \& Conquer algorithm, which combined with the improved length bound, runs in $\mathcal{O}(\operatorname{max}(n \log n \log \log n, \frac{n^2}{s}))$ time. This improves on the earlier $\mathcal{O}(\frac{n^2}{s} \log^2 n)$ result.

\begin{definition}
For $x \in A$ let $\textsc{Lis}_x$ be the $longest\ increasing\ subsequence$ of $A$ ending in $x$.
\end{definition}




\begin{definition}
For integer $x$ and sequence of numbers $A$, let $A_{\leq x}$ be the subsequence of $A$ consisted of elements that are equal of smaller than $x$. Analogously we define $A_{>x}$.
\begin{itemize}
\item $L_{\leq x}$ is a \Lis\ for $A_{\leq x}$.
\item $L_{>x}$ is a \Lis\ for $A_{>x}$.
\end{itemize}
\end{definition}

Let $A_{\ell}$ and $A_r$ be the left and right halves of $A$ respectively. Our goal is to efficiently find an element $m$ such that
concatenation of $L_{\leq m}$ for $A_\ell$ and $L_{> m}$ for $A_r$ is a \Lis\ for original input $A$. 
Having such element would allow us to recursively solve \Lis\ for $A_\ell$ and $A_r$ independently. We will show that such element can be found during a single \LisLen\ computation without changing its time and memory complexity.

\begin{definition}
For $x \in A$, let pred$(x) \in A_\ell$ be such that there is a $\textsc{Lis}_x$ in $A$
such that the last element of it belonging to $A_\ell$ is pred$(x)$, or null if the length of $\textsc{Lis}_x$ is 1 and $x \notin A_\ell$.
\end{definition}

\begin{claim} Let $a_t \in P_i$, then \\
pred$(a_t) = \begin{cases}
a_t & \text{if } a_t \in A_\ell \\
\text{null} & \text{if } a_t \in A_r \land i = 1 \\
\text{pred}(y) & \text{where } y = \operatorname{top}(P_{i-1}^{t-1})
\end{cases}$
\end{claim}

\begin{proof}
If $a_t \in A_\ell$ its trivial, so assume $a_t \in A_r$. Now, if $a_t \in P_1$, then the $\textsc{Lis}_{a_t}$ is of
length $1$, thus by definition $\operatorname{pred}(a_t) = \text{null}$. Now, assume that $a_t \in A_r \land a_t \in P_i \land i > 1$. Then  
$\textsc{Lis}_{a_t}$ is of length $i$. Since $y = \text{top}(P_{i-1}^{t-1})$ is smaller than $a_t$ (by Observation~\ref{obs:filtering}), and $a_t$ is later in $A$ than $y$, if we append $a_t$ to $\textsc{Lis}_y$ it is still a valid $increasing\ subsequence$.
Notice that since length of $\textsc{Lis}_y$ is $i-1$, by appending $a_t$ to it we obtained $\textsc{Lis}_{a_t}$. Since $a_t \in A_r$ the last element belonging to $A_\ell$ in $\textsc{Lis}_{a_t}$ is the same one that in $\textsc{Lis}_y$ making $\operatorname{pred}(a_t) = \operatorname{pred}(y)$.  
\end{proof}

It can be easily seen how to modify the \Ppass\ procedure and store the pred of an element using only $\mathcal{O}(n)$ time per \Ppass.
We can also see that to compute pred for any element that is not filtered out during a \Ppass\ we can use pred values 
and memoising the additional value only for the $\mathcal{O}(s)$ elements that we already store in memory, thus keeping both time and memory complexity with respect to big-$\mathcal{O}$ notation.
We deduce that the previously mentioned element $m$ can be $\operatorname{pred}(x)$ for any $x$ in the last pile and since computing pred values does not change the time and memory complexity of the \Ppass\ 

\begin{observation}
The time complexity for solving the \Lis\ can be written as:
\[
T(n, s) = \begin{cases}
c'n \log n \text{ if } n \leq s \\
2T(\frac{n}{2}, s) + c\operatorname{max}(n \log n, \frac{n^2}{s}) \text{ otherwise}
\end{cases}
\]
For some constants $c, c'$, which is $\mathcal{O}(\operatorname{max}(n \log n \log \log n, \frac{n^2}{s}))$.
\end{observation}
This is because we can find the element $m$ during a single \Ppass\ which runs in $\mathcal{O}(\operatorname{max}(n \log n, \frac{n^2}{s}))$ time 
and then solve on two halves of the input independently.
