\englishabstract{
A subsequence of an array is any sequence that can be derived by deleting some of the elements from the original array and keeping the rest in their relative order.    
Given an input array of comparable elements, we aim to find either a longest increasing subsequence or its length using parametrized space.
It is well known that this problem can be solved in $\mathcal{O}(n \log n)$ time without memory constraints. 
In~\cite{kiyomi2017space}, the authors presented a space-parametrized \Psort\ algorithm. Given a constraint $s$ on the number of words allowed in the working memory (where $\sqrt{n} \leq s \leq n$),
their algorithm computes the length of a LIS in $\mathcal{O}(\frac{n^2}{s} \log n)$ time. They also presented an algorithm for recovering a LIS in $\mathcal{O}(\frac{n^2}{s} \log^2 n)$ time. 
In this paper, we present an improved analysis of the space-parametrized \Psort\ algorithm introduced in~\cite{kiyomi2017space}.
We show that the length computation runs in $\mathcal{O}(\max(n \log n, \frac{n^2}{s}))$ time.
We also present an improved divide-and-conquer algorithm for recovering the sequence, which runs in $\mathcal{O}(\max(n \log n \log \log n, \frac{n^2}{s}))$ time while using $\mathcal{O}(s)$ words in the working memory.
}

\section{Introduction}

The longest increasing subsequence (LIS for short) problem is a classic algorithmic problem. 
The input is a sequence of $n$ numbers $A = (a_1, a_2, \dots, a_n)$, and our goal is either to find the length of a LIS -- 
the largest integer $k$ for which there exist indices $i_1 < i_2 < \dots < i_k$ such that $a_{i_1} < a_{i_2} < \dots < a_{i_k}$, or to extract such subsequence.

\subsection{Previous results}
Efficient algorithms for finding a LIS are well known, for example, the \Psort\ algorithm devised by Mallows \cite{mallows}, which runs in $\mathcal{O}(n \log n)$ time for a sequence of length $n$ while using $\mathcal{O}(n)$ words of working space.
Given a sequence of $n$ elements, \Psort\ assigns each element to a \textit{pile}, where each pile contains information about the longest increasing subsequence of the input that ends in that element. 
While the \Psort\ algorithm achieves the lower bound in terms of time complexity for finding a LIS \cite{lis-lb}, the question of how little memory one needs to efficiently solve the problem remains unanswered. 
A variation of \Psort\ is presented in \cite{kiyomi2017space}, which, for a given memory parameter $s$, can find the length of a LIS in $\mathcal{O}(\frac{n^2}{s} \log n)$ time and compute the actual subsequence in $\mathcal{O}(\frac{n^2}{s} \log^2 n)$ time, but only for $\sqrt{n} \leq s$.

\subsection{Our contribution}
In this paper, we improve the bounds achieved in \cite{kiyomi2017space}. For finding the length of a LIS, we conduct a tighter time analysis, showing that the algorithm proposed in \cite{kiyomi2017space} actually runs in $\mathcal{O}(\max(n \log n, \frac{n^2}{s}))$ time,
while maintaining the $\mathcal{O}(s)$ bound for the number of words in the working memory. 
Next, we observe that an alternative, simpler divide-and-conquer algorithm can be derived, which is more efficient than the one provided in \cite{kiyomi2017space}.
Combined with the tighter analysis for a LIS length computation, it runs in $\mathcal{O}(\max(n \log n \log \log n, \frac{n^2}{s}))$ time.

\subsection{Related work}
The LIS problem, in addition to its classic setting, has many heavily-researched variations.
For example, we could ask for the longest increasing subsequence only on some part of the input by making range queries \cite{range-lis}.
We can also ask to maintain a LIS in a dynamic setting, where the goal is to efficiently update the LIS length after an element is inserted, deleted, or substituted. 
For a long time, it was unknown whether sublinear update time was possible for the exact version of this problem. 
Kociumaka and Seddighin \cite{dynamic-lis} were the first to provide such an algorithm, achieving $\tilde{\mathcal{O}}(n^{2/3})$ update time. 
If we allow for a $(1+\epsilon)$-approximation, Gawrychowski and Janczewski \cite{approx-lis} showed that the LIS can be maintained in polylogarithmic time.
On the other hand, certain variants of the dynamic LIS problem are conjectured to be hard; for instance, weighted dynamic LIS and range dynamic LIS have conditional lower bounds of $\Omega(n^{1/2})$ and $\Omega(n^{1/3})$, respectively \cite{dynamic-lis-lb}.
Another interesting question is the approximation of a LIS in any of these settings \cite{approx-lis, approx-lcs}. 

\subsection{Applications}
As one of the most classic algorithmic problems, a LIS problem has a variety of applications across diverse scientific domains.
In bioinformatics, a LIS is a fundamental tool for genome rearrangement analysis and sequence alignment, where it is often used to find maximum weight traces or to solve the Longest Common Subsequence (LCS) problem for some special cases \cite{delcher1999alignment}. 
It also plays a crucial role in statistical physics and probability theory; notably, the distribution of the length of a LIS in random permutations is a central topic in the study of random matrices and the Tracy-Widom distribution \cite{baik1999distribution}. 
Additionally, LIS algorithms are used in data analysis for identifying trends in time-series data and in version control systems for sequence differencing.
A notable example is the \textit{Patience Diff} algorithm \cite{patience-diff}, which can be used for more readable output while using the \texttt{diff} with \textit{--patience} flag.

\section{Preliminaries}

We define two variants of the Longest Increasing Subsequence problem.

\problembox
{Longest increasing Subsequence (\Lis)}
{A sequence of $n$ numbers $A = (a_1, a_2, \dots, a_n)$.}
{The longest subsequence $(a_{i_1}, a_{i_2}, \dots, a_{i_k})$ of $A$ such that $i_1 < i_2 < \dots < i_k$ and $a_{i_1} < a_{i_2} < \dots < a_{i_k}$.}

\problembox
{Length of \Lis \ (\LisLen)}
{A sequence of $n$ numbers $A = (a_1, a_2, \dots, a_n)$.}
{$k$ -- the largest integer such that there exist an increasing subsequence in $A$ of length $k$}


We use the RAM model, where the input sequence is read-only, and the working memory is limited to $\mathcal{O}(s)$ words of $\mathcal{O}(\log n)$ bits.

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%

\subsection{\Psort}

Our algorithm is based on the classic \Psort. Let us define basic notions used throughout the algorithm.
The idea of the algorithm is to maintain \textit{piles} -- stacks of integers such that for all $x \in A$, $x$ belongs to the $i$-th pile if and only if the \textit{longest increasing subsequence} of $A$ ending in $x$ has length $i$.
By this definition we can see that each element $x$ belongs to exactly one pile.
Since both classic and space-parametrized variants of the \Psort\ algorithm are based on sequential passes through the input array, we define
pile operations with respect to the time when they are being used.

\begin{definition}
For integers $i, t$, we refer to the state of the $i$-th pile after reading the first $t$ elements of the input sequence as $P_{i}^{t}$, and $P_i$ will represent the $i$-th pile after reading the whole input sequence.
\end{definition}
Now we define some useful operations that can be conducted on piles.
Let $P$ be some state of the pile (for example $P_3^4$ or $P_2$), then:
\begin{itemize}
\item $|P|$ is the number of elements that belong to $P$
\item $\operatorname{top}(P)$ refers to the element on the top of the pile (the one that was added most recently to it), or null if $P$ is empty.
\item For an integer $j$, $P[j]$ is the $j$-th element of $P$, where $\operatorname{top}(P) = P[|P|]$, and $P[1]$ is the element on the bottom of $P$. 
\end{itemize}

It follows from the pile definition that if we are solving the \LisLen\ problem, then our goal is to compute the largest integer $k$ such that $P_k$ is non-empty.
The classic \Psort\ algorithm achieves that by constructing all piles during a single pass through the input sequence. When reading the $t$-th element of $A$, $a_t$, we find
the smallest $i$ such that $\operatorname{top}(P_i^{t-1}) \geq a_t$ and we push $a_t$ to the top of the $i$-th pile, or we create a new, rightmost pile with a single element $a_t$.
Since each element belongs to exactly one pile, for every $t$, if $a_t$ is in the $i$-th pile then $P_i^t$ is $P_i^{t-1}$ with $a_t$ pushed onto it, and $P_j^t = P_j^{t-1}$ for all $j \neq i$.
Since copying and storing the pile states at every step would introduce time and memory issues, and the state transitions are straightforward, we only maintain their current state (in the moment $t$ after reading $a_t$).
Thus, we omit the upper indices in the implementation of \Psort\ algorithm below.

\begin{algorithm}
    \caption{\Psort}
    \label{alg:psort}
    \begin{algorithmic}[1]
        \State initialize $P_{0}$ with the single element $-\infty$
        \State set $k := 0$ \Comment{\textcolor{gray}{$k$ will be the index of the rightmost non-empty pile}}
        \For{$a_t \in A$}
            \If{$a_t > \text{top} (P_{k})$}
                \State create an empty pile -- $P_{k+1}$, and push $a_t$ to $P_{k+1}$
                \State set $k := k + 1$
            \Else
                \State set $h$ to be the smallest index with $x \leq \operatorname{top}(P_h)$, and push $x$ to $P_h$
            \EndIf
        \EndFor
        \State \Return $k$
  \end{algorithmic}
\end{algorithm}


\begin{observation}\label{obs:pile}(\cite{burnstein})
Algorithm~\ref{alg:psort} correctly computes piles, in other words: \\
Element $a_t \in P_i$ if and only if a \textit{longest increasing subsequence} of $A$ ending in $a_t$ is of length $i$
\end{observation}

\subsection{Space-parametrized \Psort}
Now we define the space-parametrized variant of \Psort\ that originates from \cite{kiyomi2017space}.

Assume that our working memory is limited to $s'$ words. Let $s := \frac{s'}{4}$ and assume that $\sqrt{n} \leq s \leq n$.
Similarly to the classic \Psort\ we will process the input array sequentially while maintaining piles. However, since our memory is limited and we cannot afford
to keep all piles in the memory, we will pass through the input multiple times. Each time we will store only one, chosen pile and top elements of some other piles. The idea is clearly presented in \cite{kiyomi2017space} (Lemma 4.2, Lemma 4.3).

More precisely, now we define a procedure \Ppass\ that, given the $i$-th pile $P_i$, performs two passes through the input array.
During the first pass it computes an index $j$ such that $i + s \leq j \leq i + 2s \land |P_j| \leq s$ or $j$ is the length of a \Lis. 
During the second pass, now with given $P_i$ and $j$ it will compute whole $P_j$, or only the $\operatorname{top}(P_j)$ if $j$ is the length of a \Lis.

\begin{observation}\label{obs:filtering} (\cite{kiyomi2017space}, Observation 4.1)
Suppose we are reading $a_t$, and $a_t \in P_j$. Let $i$ be a fixed integer. \\
If $j \neq i$, then:
\begin{itemize}
\item $a_t < \operatorname{top}(P_i^{t-1})$ if $j < i$
\item $a_t > \operatorname{top}(P_i^{t-1})$ otherwise
\end{itemize} 
\end{observation}

Recall that in the \Psort\ algorithm, for an element $a_t$, to determine to which pile it belongs, it suffices to know only the values of the top elements of the piles at step $t-1$. 
Moreover, by Observation~\ref{obs:filtering}, if we want to check whether this index -- call it $k$, belongs to a specific interval -- in our case, whether $i \leq k \leq i + 2s$---it suffices to know only the top elements of the piles whose indices are also in this interval.
Below, we define the \textit{filtering} method for the \Ppass\ procedure.
Suppose we are processing $a_t$. We can encounter one of the following three cases: 
\begin{enumerate}
\item If $a_t = \operatorname{top}(P_i^t)$, then $k = i$.
\item Otherwise, if $a_t < \operatorname{top}(P_i^{t-1})$, then, by Observation~\ref{obs:filtering}, $k < i$, and we ignore $a_t$.
\item Otherwise, $a_t > \operatorname{top}(P_i^{t-1})$, and again, by Observation~\ref{obs:filtering}, $k > i$.
Let $r := \operatorname{top}(P_{i + 2s}^{t-1})$. Then:
    \begin{enumerate}
    \item If $r$ exists and $a_t > r$, then $k > i + 2s$, and we ignore $a_t$.
    \item Otherwise, if $r$ is undefined or $a_t \leq r$, we have $i < k \leq i + 2s$, and $k$ is the smallest index such that $a_t \leq \operatorname{top}(P_k^{t-1})$.
    \end{enumerate}
\end{enumerate}
Observe that if the first case is false, then $\operatorname{top}(P_i^{t-1}) = \operatorname{top}(P_i^{t})$; thus, exactly one of these cases holds.

Notice that if we have the whole $P_i$ in memory, we can process the first two cases in constant time by maintaining a pointer to the most recently read element $h := \operatorname{top}(P_i^{t-1})$ in $P_i$.
In the first case, we compare $a_t$ to the element in $P_i$ that is after $h$. We can determine whether we are in the second or third case by comparing $a_t$ and $h$.

It is easy to see that if we maintain $\operatorname{top}(P_k^{t-1})$ for all $i < k \leq i + 2s$, then we can distinguish cases 3(a) and 3(b) in constant time and find the value of $k$ in 3(b) in $\mathcal{O}(\log s)$ time.
We call the search in 3(b) an \textit{expensive filter}. Below, we present the \Ppass\ implementation (Algorithm 3 in \cite{kiyomi2017space}).

\begin{algorithm}
    \caption{First pass in \Ppass, assume $P_i$ is given, and $|P_i| < s$}
    \begin{algorithmic}[1]
    \State set $h := 0$ \Comment{\textcolor{gray}{$h$ points to the most recently read element in $P_{i}$}}
    \State set $r := i$ \Comment{\textcolor{gray}{Is the index of the current rightmost pile}}
    \State set $p_{i+1}, p_{i+2}, \ldots, p_{i+2s}$ to $\infty$ \Comment{\textcolor{gray}{$p_k$ is the current $\operatorname{top}(P_{k})$}}
    \State set $c_{i+1}, c_{i+2}, \ldots, c_{i+2s}$ to $0$ \Comment{\textcolor{gray}{$c_k$ is the current size of $P_k$}}
    \For{$ x \in A$} 

        \If{$x = P_i[h+1]$} \Comment{\textcolor{gray}{$x$ is the next element in $P_i$}}
            \State set $h := h + 1$ 
            \State continue
        \ElsIf {$x < P_i[h]$ or $x > p_{i+2s}$} \Comment{\textcolor{gray}{$x$ belongs to the $P_k$ where $k < i \vee k > i+2s$}}
            \State continue
        \EndIf

        \If{$x > p_r$}
            \State set $r := r + 1$ and $k := r$
        \Else
            \State set $k$ to be the smallest such that $x < p_k$
        \EndIf
            
        \State set $p_k := x$ and $c_k := c_k + 1$
    \EndFor
    
    \State set $j$ to be the biggest, such that $0 < c_j \leq s$
    \State \Return $j$
    \end{algorithmic}
\end{algorithm}

After this pass, if $r < i + 2s$, we can conclude that $r$ is the length of a \Lis, since no element belongs to a pile with a larger index.
Then, we can set $j := r$ and return $j$ as the \LisLen, along with $p_j = \operatorname{top}(P_j)$. 
It is easy to see how to modify the procedure to compute the whole $P_j$ if $j$ is given: we can maintain an additional stack and push the elements that belong to the $j$-th pile onto it.

\begin{observation} \label{obs:ppass-advance}
Since $s > \sqrt{n}$ there are at most $\frac{n}{s} \leq s$ piles with more than $s$ elements which implies that $j > i + s$.
\end{observation}

\begin{observation} \label{obs:ppass-time}
The time complexity of a single \Ppass\ is $\mathcal{O}(n + W \log s )$, where $W$ is the number of calls to the expensive filter during the procedure.
\end{observation}

\section{New results}

The complexity analysis for the space-parametrized \Psort\ in \cite{kiyomi2017space} for finding \LisLen\ yielded an $\mathcal{O}(\frac{n^2}{s} \log n)$ time.
By providing a tighter estimation of the number of calls to the \textit{expensive filter} procedure, we prove that this algorithm actually runs in $\mathcal{O}(\max(n \log n, \frac{n^2}{s}))$ time.
Furthermore, \cite{kiyomi2017space} presented a divide-and-conquer algorithm for computing a \Lis\ in $\mathcal{O}(\frac{n^2}{s} \log^2 n)$ time.
We provide a simpler and more efficient divide-and-conquer strategy that, combined with the improved bound for computing the \LisLen, runs in $\mathcal{O}(\max(n \log n \log \log n, \frac{n^2}{s}))$ time.


\subsection{Length computation}
We provide a tighter time complexity analysis of the \Psort\ algorithm presented in \cite{kiyomi2017space}. 
We show that the algorithm runs in $\mathcal{O}(\max(n \log n, \frac{n^2}{s}))$ time, improving upon the previous $\mathcal{O}(\frac{n^2}{s} \log n)$ bound.

\begin{definition}
    Let $j_i$ be the index of the pile kept in memory after $i$-th \Ppass\ and we assume $j_0=0$.
\end{definition}

\begin{definition}
Let $W_i$ be the number of \textit{expensive filter} operations during the $i$-th \Ppass.
\end{definition}

\begin{definition}
Let $K$ be the total number of \Ppass -es.
\end{definition}

By using Observation~\ref{obs:ppass-time} we can see that time complexity of space-parametrized \Psort\ for $\sqrt{n} \leq s \leq n$ is:
\[
T(n,s) \in \mathcal{O}(nK + \log s \sum_{i=1}^K W_i).
\]

\begin{claim} \label{claim:linear-filters}
The total number of expensive filter operations is linear, i.e.,
\[
\sum_{i=1}^K W_i \in \mathcal{O}(n).
\]
\end{claim}

\begin{proof}
For each pass $i \in \{1, \dots, K\}$, let us define the active interval $I_i = (j_{i-1}, j_{i-1}+2s]$.

Fix a pass $i$. During the first phase of this pass, the \textit{expensive filter} can only be invoked only for some elements whose pile index belongs to $I_i$.
Once the new boundary $j_i$ is determined, the second phase can invoke the \textit{expensive filter} only if the pile index is in $(j_{i-1}, j_i]$. 
By \Ppass\ construction, we know that $j_i \leq j_{i-1} + 2s$, which means $(j_{i-1}, j_i] \subseteq I_i$. Thus, we can see that:
\[
W_i \le \sum_{k \in I_i} |P_k| + \sum_{k=j_{i-1}+1}^{j_i} |P_k| \leq 2\sum_{k \in I_i} |P_k|.
\]
This gives us: 
\[
\sum_{i=1}^K W_i \le 2\sum_{i=1}^K \sum_{k \in I_i} |P_k|.
\]

By \Ppass\ construction (Observation~\ref{obs:ppass-advance}) we have $j_i \ge j_{i-1}+s$  or $j_i$ is the length of \Lis.
This implies that the left endpoints of the intervals $I_i$ advance by at least $s$ for all but the last pass.
Observe that (when not counting the last pass) each interval $I_i$ has a width of exactly $2s$, any pile index $k$ can belong to at most two intervals $I$.
Therefore, summing the pile sizes over all intervals yields at most twice the total number of elements:
\[
\sum_{i=1}^{K-1} \sum_{k \in I_i} |P_k| + \sum_{k \in I_K} |P_k| \le 2\sum_{k=1}^n |P_k| + \sum_{k=1}^n |P_k|  = 3n.
\]
Thus, we get $\sum_{i=1}^K W_i \le 6n \in \mathcal{O}(n)$.
\end{proof}

Observe that $K \leq \frac{n}{s}$. 
This is because after each \Ppass\ the index of the computed pile increases by at least $s$, and
% This is because after each \Ppass\ we compute a pile with an index at least $s$ larger than that of the previous one. 
the algorithm terminates when it reaches a pile equal to the length of a LIS which is at most $n$.
Thus, the total time complexity is:
\[
T(n,s) \leq cn \frac{n}{s} + c'n\log s
\]
for some constants $c, c'$. It is easy to see that $T(n, s) \in \mathcal{O}(\max(n\log n, \frac{n^2}{s}))$ as desired.

\subsection{LIS computation}
To compute the \Lis, we present an alternative divide-and-conquer algorithm, which combined with the improved length bound, runs in $\mathcal{O}(\max(n \log n \log \log n, \frac{n^2}{s}))$ time. This improves on the earlier $\mathcal{O}(\frac{n^2}{s} \log^2 n)$ result.

\begin{definition}
For $x \in A$ let $\textsc{Lis}_x$ be any one of the \textit{longest increasing subsequence} of $A$ ending in $x$.
\end{definition}

\begin{definition}
For an integer $x$ and sequence of numbers $A$, let $A_{\leq x}$ be the subsequence of $A$ consisting of elements that are equal of smaller than $x$. Analogously we define $A_{>x}$.
\end{definition}

\begin{definition}
Then, we define:
\begin{itemize}
\item $L_{\leq x}$ is a \Lis\ for $A_{\leq x}$.
\item $L_{>x}$ is a \Lis\ for $A_{>x}$.
\end{itemize}
\end{definition}

Let $A_{\ell}$ and $A_r$ be the left and right halves of $A$ respectively. Our goal is to efficiently find an element $m$ such that
concatenation of $L_{\leq m}$ for $A_\ell$ and $L_{> m}$ for $A_r$ is a \Lis\ for original input $A$. 
Having such element would allow us to recursively solve \Lis\ for $A_\ell$ and $A_r$ independently.
We will show that such element can be found during a single \LisLen\ computation without changing its time and memory complexity.

\begin{definition}
For $x \in A$, let $\operatorname{pred}(x) \in A_\ell$ be such that there is a $\textsc{Lis}_x$ in $A$
such that its last element belonging to $A_\ell$ is pred$(x)$, or null if the length of $\textsc{Lis}_x$ is 1 and $x \notin A_\ell$.
\end{definition}

\begin{claim} Let $a_t \in P_i$, then \\
pred$(a_t) = \begin{cases}
a_t & \text{if } a_t \in A_\ell \\
\text{null} & \text{if } a_t \in A_r \land i = 1 \\
\operatorname{pred}(\operatorname{top}(P_{i-1}^{t-1})) & \text{if } a_t \in A_r \land i > 1
\end{cases}$
\end{claim}

\begin{proof}
Case when $a_t \in A_\ell$ is trivial. Assume $a_t \in A_r$. Now, if $a_t \in P_1$, then the $\textsc{Lis}_{a_t}$ is of
length $1$, thus by definition $\operatorname{pred}(a_t) = \text{null}$. Now, assume that $a_t \in A_r \land a_t \in P_i \land i > 1$. Then  
$\textsc{Lis}_{a_t}$ is of length $i$. Since $y := \text{top}(P_{i-1}^{t-1})$ is smaller than $a_t$ (by Observation~\ref{obs:filtering}), and $a_t$ is later in $A$ than $y$, if we append $a_t$ to $\textsc{Lis}_y$ it is still a valid $increasing\ subsequence$.
Notice that since length of $\textsc{Lis}_y$ is $i-1$, by appending $a_t$ to it we obtain $\textsc{Lis}_{a_t}$. Since $a_t \in A_r$ the last element belonging to $A_\ell$ in $\textsc{Lis}_{a_t}$ is the same one that in $\textsc{Lis}_y$ making $\operatorname{pred}(a_t) = \operatorname{pred}(y)$.  
\end{proof}

\begin{observation}
Let $a \in A$ be an element that belongs to the last pile. 
Then, for $m := \operatorname{pred}(a)$ concatenation of $L_{\leq m}$ for $A_\ell$ and $L_{> m}$ for $A_r$ is a \Lis of $A$.
\end{observation}

\begin{proof}
This follows directly from the definition of $\operatorname{pred}$. Let us choose an element $a$ from the last pile. 
Consider a $\textsc{Lis}_a$ where $m$ is the last element from $A_\ell$ belonging to this sequence.
Since $a$ is located in the last pile, $\textsc{Lis}_a$ is a \Lis of $A$.
Because $\textsc{Lis}_a$ is an increasing sequence and $m$ is in it, there exist increasing subsequences $L_1 \text{ and } L_2$, such that $\textsc{Lis}_a$ is their concatenation and $m$ is the last element of $L_1$.
It immediately follows that each element of $L_1$ is less than or equal to $m$ and each element of $L_2$ is greater than $m$.
Since $m \in A_\ell$, $L_1$ is a subsequence of $A_\ell$. Because $m$ is the last element from $A_\ell$ in $\textsc{Lis}_a$, $L_2$ is a subsequence of $A_r$.
Therefore, since $\textsc{Lis}_a$ is a \Lis of $A$, $L_1$ is $L_{\leq m}$ and $L_2$ is $L_{> m}$.
\end{proof}

It is easy to see how to modify the \Ppass\ procedure to store the $\operatorname{pred}$ of every element, adding only $\mathcal{O}(n)$ time per \Ppass.
We can also see that to compute the $\operatorname{pred}$ for any element that is not filtered out during a \Ppass, we can use the existing $\operatorname{pred}$ values 
and store the additional value only for the $\mathcal{O}(s)$ elements that we already keep in memory, thus preserving both the time and memory complexity bounds.
We deduce that $m$ is $\operatorname{pred}(x)$ for some $x$ in the last pile. Since computing the $\operatorname{pred}$ values does not change the time and memory complexity of the \Ppass\ procedure, we obtain the following observation.

\begin{observation}The time complexity for solving the \Lis\ is:
$$T(n, s) = \begin{cases}
c'n \log n \text{ if } n \leq s \\
2T(\frac{n}{2}, s) + c\operatorname{max}(n \log n, \frac{n^2}{s}) \text{ otherwise}
\end{cases}$$
For some constants $c, c'$. This is $\mathcal{O}(\operatorname{max}(n \log n \log \log n, \frac{n^2}{s}))$.
\end{observation}
\begin{proof}
This is because we find the element $m$ during a single \Ppass\ which runs in $\mathcal{O}(\operatorname{max}(n \log n, \frac{n^2}{s}))$ time and then solve on two halves of the input independently.

To justify the complexity bound, suppose that we claim $T(n, s) = 2T(\frac{n}{2}, s) + c_1n \log n + c_2\frac{n^2}{s}$ when $n > s$ for some constants $c_1, c_2$.
Notice that $n \log n$ dominates over $\frac{n^2}{s}$ when $s > \frac{n}{\log n}$, and there are at most $\log \log n$ such levels until the base case $n \leq s$ is reached, contributing at most $\mathcal{O}(n \log n \log \log n)$ to the total work.
Otherwise, when $\frac{n^2}{s}$ dominates, the recursion yields $\mathcal{O}(\frac{n^2}{s})$. 
Therefore, the final time complexity is $\mathcal{O}(\operatorname{max}(n \log n \log \log n, \frac{n^2}{s}))$.
\end{proof}

Below we provide the algorithm for computing \Lis\ in pseudocode.
First we define a slightly modified version of \Ppass\ that will allow us to compute $L_{\leq x}$ and $L_{> x}$ for some $x$ by ignoring elements bigger than $x$ or elements smaller than or equal to $x$ respectively.
We will assume that if we are storing an element in working memory, then we also store its $\operatorname{pred}$.
\begin{algorithm}
    \caption{\Ppass\ -- the second pass through an array, computing $P_j$}
    \label{alg:ppas-mod}
    \begin{algorithmic}[1]
        \Require sequence $A$, space limit $s$, pile $P_i$, value $lb$, value $ub$, value of pile we are computing $j$

        \State set $h := 0$ 
        \State set $r := i$
        \State set $p_{i+1}, p_{i+2}, \ldots, p_{j}$ to $\infty$ 
        \State create an empty pile $P_j$
        \For{$a_t \in A$} 

            \If{$a_t$ is $P_i[h+1]$} 
                \State set $h := h + 1$ 
                \State continue
            \ElsIf {$a_t < P_i[h]$ or $a_t > p_j$ or $a_t > ub$ or $a_t \leq lb$} 
                \State continue
            \EndIf

            \If{$a_t > p_r$}
                \State set $r := r + 1$ and $k := r$
            \Else
                \State set $k$ to be the smallest such that $a_t < p_k$
            \EndIf
            
            \If{$t \leq \frac{|A|}{2}$}
                \State $\operatorname{pred}(a_t) := a_t$
            \ElsIf{$k = 1$}
                \State $\operatorname{pred}(a_t) := null$
            \Else 
                \State $\operatorname{pred}(a_t) := \operatorname{pred}(\operatorname{top}(P_{k-1}))$
            \EndIf

            \State set $p_k := a_t$

            \If{$k = j$}
                \State push $a_t$ onto $P_j$
            \EndIf
        \EndFor

    \Return $P_j$
    \end{algorithmic}
\end{algorithm}

Now, using this modified \Ppass\ procedure, we define the \LisLen\ procedure. It returns the $\operatorname{pred}$ value of an element from the last pile, which is the splitting value $m$ in our divide-and-conquer strategy.

\begin{algorithm}[H]
    \caption{\LisLen}
    \label{alg:lislen}
    \begin{algorithmic}[1]
        \Require sequence $A$, space limit $s$, values $lb, ub$
        \State set $i := 0$ and make an empty pile -- $P_0$
        \Repeat
            \State $j := \text{run first pass of \Ppass\ with } P_i, lb, ub$ \Comment{\textcolor{gray}{Find the next reachable pile $j$}}
            \State $P_j := \text{run second pass of \Ppass\ with } P_i, j, lb, ub$ 
            \State set $i := j$ and $P_i := P_j$ 
        \Until{$i$ is the length of \Lis}
        \State \Return $(i, \operatorname{pred}(\operatorname{top}(P_i)))$ \Comment{\textcolor{gray}{Return length and the element $m$}}
    \end{algorithmic}
\end{algorithm}

Finally, we present the main divide-and-conquer algorithm for extracting the \Lis. 
When the input sequence is small enough to fit in memory, we use the classic \Psort. 
Otherwise, we use \LisLen\ to find the element $m$, recursively retrieve the $L_{\leq m}$ for $A_\ell$ and $L_{>m}$ for $A_r$, and return their concatenation.
Initially we call the procedure with values $lb, ub$ set to $-\infty$ and $\infty$ respectively.

\begin{algorithm}[H]
    \caption{\Lis}
    \label{alg:lis-dc}
    \begin{algorithmic}[1]
        \Require sequence $A$, space limit $s$, values $lb, ub$
        \If{$|A| \leq s$} \Comment{\textcolor{gray}{Whole array fits in the working memory}}
            \State \Return classic \Lis\ of $A$ within $(lb, ub)$
        \EndIf
        
        \State $(k, m) := \LisLen(A, s, lb, ub)$ \Comment{\textcolor{gray}{Find length $k$ and splitting element $m$}}
        \State split $A$ into $A_\ell, A_r$ \Comment{\textcolor{gray}{Divide input into two halves}}
        
        \If{$m = null$} \Comment{\textcolor{gray}{The \Lis\ does not contain any elements from $A_\ell$}}
            \State \Return $\Lis(A_r, s, lb, ub)$
        \Else \Comment{\textcolor{gray}{Recursively solve for both halves using $m$ as a bound}}
            \State \Return $\Lis(A_\ell, s, lb, m) + \Lis(A_r, s, m, ub)$ \Comment{\textcolor{gray}{$+$ is concatenation}}
        \EndIf
    \end{algorithmic}
\end{algorithm}

\section{Algorithm performance results}
We provide an implementation of the described algorithm in a GitHub repository\footnote{\url{https://github.com/m4jkiuwr/lis-small-space}}. 
The code is written in C++20. It uses modern C++ features like the \texttt{std::span} for clean and efficient sub-array handling and \texttt{std::optional} for representing the $\operatorname{pred}$ values.

\subsection{Implementation details}

\subsubsection{\Ppass}
The \Ppass\ procedure is implemented in two phases: \texttt{Ppass\_first} and \texttt{Ppass\_second}. 
In \texttt{Ppass\_first}, we identify the next pile $j$ such that its size $|P_j| \leq s$, while advancing the lower bound of the pile indices by at least $s$. 
We maintain the top elements of $2s$ piles in an array \texttt{p}. The binary search for the expensive filter is implemented using \texttt{std::lower\_bound} on the indices of top elements, which yields the $\mathcal{O}(\log s)$ overhead per filter operation.

In \texttt{Ppass\_second}, we reconstruct the $j$-th pile. To handle the divide-and-conquer requirements, each element in the pile is stored alongside its predecessor index or value. 
If the current element $a_t$ is in the first half of the current range, its predecessor is set to itself. 
Otherwise, it inherits the predecessor of the top of the $(k-1)$-th pile at time $t$.

\subsubsection{Procedure for finding the \LisLen}
The procedure \texttt{LisLen} repeatedly calls the \Ppass\ procedure.
It maintains the \textit{current pile} in memory (we put an \texttt{assert} statement to check that this pile has at most $s$ elements) and iteratively advances to the next reachable pile $j$ until the last pile of the LIS is found. 
The total length is accumulated across these jumps. 
Crucially, it returns both the length of the LIS and the splitting element $m$ (the predecessor of an element in the last pile), which is essential for the recursion.

\subsubsection{Procedure for computing a \Lis}
The \texttt{Lis} function implements the divide-and-conquer strategy. 
If the input size is less than or equal to $s$, it falls back to \texttt{classicLIS}, which uses the standard $O(n \log n)$ patience sorting algorithm with $O(n)$ space. 
For larger inputs, it calls \texttt{LisLen} to find the split point $m$ and then recursively solves for $A_\ell$ and $A_r$. 
We use \texttt{std::span} to avoid copying sub-arrays during recursion, ensuring that only $O(s)$ additional words are used for storing the piles and metadata.
Notice that although the algorithm uses recursion, the only value created in each call is the $m$ value for splitting the array and two pointers identifying the \texttt{std::span} boundaries.
Since recursion depth is at most $\log n$, the memory overhead created by it is $\mathcal{O}(\log n)$ words.

\subsection{Testing}

To evaluate the performance and verify the correctness of our implementation, we provide a \texttt{python} script \texttt{test\_lis.py}.
Since our main contribution is the new divide-and-conquer procedure, we focused on testing the procedure that computes the whole LIS sequence.
\subsubsection{Generating the test cases}
For generating pseudo-random tests we use the built-in \texttt{python} \texttt{random} library.

\section{Summary}
In this paper, we revisited the space-parametrized variant of \Psort\ from~\cite{kiyomi2017space}.
By bounding the total number of \textit{expensive filter} operations across all passes, we showed that computing \LisLen\ runs in
$\mathcal{O}(\max(n \log n, \frac{n^2}{s}))$ time for $\sqrt{n} \leq s \leq n$, improving the previously stated $\mathcal{O}(\frac{n^2}{s} \log n)$ bound.
We then derived a simpler divide-and-conquer reconstruction strategy which, using the same \Ppass\ procedure, computes an actual \Lis\ in
$\mathcal{O}(\max(n \log n \log \log n, \frac{n^2}{s}))$ time while preserving the $\mathcal{O}(s)$ bound on the number of words in the working space.
Finally, we implemented the algorithms in C++20 and benchmarked the implementation by measuring time and memory usage when running tests.

\subsubsection{Open questions}
A natural open question is whether there exists an exact algorithm for \LisLen\ and \Lis\ using $o(\sqrt{n})$ words of working memory.
As pointed out in~\cite{kiyomi2017space}, a definite 'no' answer would be surprising as it implies that \textsc{SC} $\neq$ \textsc{P} $\cap$ \textsc{PolyL},
where \textsc{SC} -- Steve's Class is the class of problems solvable by an algorithm running in polynomial time and polylogarithmic space ~\cite{sc-cfl} ~\cite{rl-sc}.
One could ask about the existence of a log-space algorithm. Showing NL-hardness of (a decision version of) LIS would provide some evidence for a 'no' answer to this question.
